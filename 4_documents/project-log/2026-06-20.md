### 2. SCSS 빌드 체계 도입

기존 프론트는 일반 CSS와 React 인라인 스타일을 사용하고 있었으며 Sass 컴파일 환경은 없었다. 별도 프론트 프로토타입의 `tokens.scss` 구조를 검토한 뒤, 최신 헤더와 지도 스타일을 잃지 않도록 현재 CSS를 기준으로 SCSS 토큰 구조를 도입했다.

변경한 주요 파일:

- `frontend/src/styles/tokens.css` → `frontend/src/styles/tokens.scss`
- `frontend/src/main.jsx`
- `frontend/package.json`
- `frontend/pnpm-lock.yaml`
- `frontend/pnpm-workspace.yaml`

구현 내용:

- 브랜드, 중립색, 위험 상태, 반경, 그림자를 SCSS map으로 정의했다.
- `@each`를 이용해 SCSS map에서 CSS custom property를 생성한다.
- 위험 상태 목록은 `sass:list`의 `list.nth()`를 사용해 Dart Sass의 전역 함수 폐기 경고를 제거했다.
- 기존 최신 헤더, 카카오 로그인, 지도 레이아웃 CSS는 그대로 유지했다.
- `main.jsx`가 `tokens.scss`를 직접 import하도록 변경했다.
- `sass 1.101.0`을 개발 의존성으로 추가했다.

pnpm 11에서 기존 `package.json`의 `pnpm.onlyBuiltDependencies` 설정이 더 이상 적용되지 않아 `pnpm-workspace.yaml`의 `allowBuilds`로 이전했다.

```yaml
allowBuilds:
  '@parcel/watcher': true
  esbuild: true
```

저장소 내부 `.pnpm-store`와 프로젝트 파일이 하드링크되는 문제를 방지하기 위해 이후 설치는 저장소 밖의 임시 store를 사용했다.

```powershell
corepack pnpm install --no-frozen-lockfile --store-dir C:\Tmp\zipt-pnpm-store
```

### 3. 프론트엔드 구조 비교 및 방향 결정

백엔드 연동 테스트용으로 만들어진 `C:/KOSTA_Projects/front_zipt1`과 현재 프론트를 비교했다.

비교 결과:

- `front_zipt1/src`의 파일 45개 중 35개가 빈 파일이었다.
- 실제 활성 라우트는 홈(`/`) 하나였다.
- React Query Provider와 대부분의 페이지 라우트가 주석 처리돼 있었다.
- `node_modules` 파일 1,482개가 Git 추적 대상에 포함돼 있었다.
- 현재 프로젝트에는 지도, OAuth, API 계층, 마이페이지, 용어 사전, 분석 흐름이 이미 구현돼 있었다.
- 두 프로젝트는 React/Vite 버전, 패키지 관리자, 디자인 토큰 방향이 달랐다.

따라서 `front_zipt1`의 미완성 구조에 현재 화면을 맞추지 않고, 현재 `frontend`를 공식 프론트로 유지하면서 `pages`, `features`, `shared`, `layouts`, `app` 구조를 점진적으로 도입하기로 결정했다. `front_zipt1`은 백엔드 API 연동 예제 또는 테스트 클라이언트로만 참고한다.

### 4. React Router와 App 계층 도입

기존 앱은 `legacy/app.jsx` 내부의 문자열 상태로 현재 화면을 관리했다. 이를 React Router 기반 URL 라우팅으로 전환했다.

추가한 파일:

- `frontend/src/app/App.jsx`
- `frontend/src/app/router.jsx`
- `frontend/src/app/routes.js`

추가한 의존성:

```json
"react-router-dom": "^6.30.1"
```

라우트 매핑:

| 기능 | URL |
| --- | --- |
| 홈 | `/` |
| 등기부 분석 | `/analysis` |
| 보증보험 확인 | `/insurance` |
| 용어 정리ZIP | `/glossary` |
| 인프라 브리핑 | `/map` |
| 매물 비교 | `/compare` |
| 마이페이지 | `/mypage` |
| OAuth 콜백 | `/oauth/callback` |

기존 `onGo("home")` 방식의 화면 이동 인터페이스는 유지하면서 내부 구현을 `useNavigate()`로 교체했다. 이에 따라 메뉴 이동 시 실제 URL이 변경되고 직접 URL로 진입할 수 있게 됐다.

### 5. Header, Footer, MainLayout 분리

기존 `legacy/app.jsx`에 함께 있던 상단 내비게이션과 페이지 골격을 Layout 계층으로 분리했다.

추가한 파일:

- `frontend/src/layouts/Header.jsx`
- `frontend/src/layouts/Footer.jsx`
- `frontend/src/layouts/MainLayout.jsx`

구현 내용:

- `Header.jsx`에서 로고, 메뉴, 로그인, 프로필, 로그아웃 UI를 관리한다.
- `Footer.jsx`에 서비스 소개, 용어 정리ZIP, 인프라 브리핑 링크와 저작권 문구를 추가했다.
- `MainLayout.jsx`가 Header, 본문, Footer의 공통 골격을 담당한다.
- 앱 shell을 flex column 구조로 변경해 콘텐츠가 짧아도 Footer가 화면 아래에 배치되도록 했다.
- 모바일 화면에서는 Footer 항목이 세로로 배치되도록 반응형 SCSS를 추가했다.

### 6. 페이지를 `legacy` 밖으로 이동

화면 구현 파일을 URL 페이지 단위로 `src/pages` 아래에 배치했다.

이동 결과:

```text
src/pages/
├─ analysis/AnalysisPage.jsx
├─ glossary/GlossaryPage.jsx
├─ home/HomePage.jsx
├─ insurance/InsurancePage.jsx
├─ map/MapPage.jsx
└─ mypage/MyPage.jsx
```

각 페이지는 더 이상 `window.Home`, `window.Terms`, `window.MyPage`처럼 전역 객체에 자신을 등록하지 않는다. `App`과 페이지 조립 코드가 일반 ES module import로 각 페이지를 불러온다.

페이지에서 사용하던 React hook도 전역 `React` 객체가 아니라 다음과 같이 직접 import하도록 변경했다.

```javascript
import { useEffect, useRef, useState } from "react";
```

### 8. 페이지 API와 상태 로직을 Feature 계층으로 이동

페이지가 화면 조립에 집중하도록 데이터 선택, API 요청, 계산, 복합 상태 변경을 기능별 hook/service로 분리했다.

추가한 파일:

- `frontend/src/features/home/service.js`
- `frontend/src/features/analysis/service.js`
- `frontend/src/features/insurance/useInsuranceSimulator.js`
- `frontend/src/features/glossary/useGlossary.js`
- `frontend/src/features/map/service.js`
- `frontend/src/features/mypage/useReportLibrary.js`
- `frontend/src/shared/api/client.js`

역할:

- `home/service.js`: 홈 대시보드에 필요한 사용자, 계약, 매물, 일정 데이터를 제공한다.
- `analysis/service.js`: 데모 등기부 분석 리포트를 제공한다.
- `useInsuranceSimulator.js`: 보증금 변경, LTV, 선순위 채권 비율, 가입 가능 여부, 최대 허용 보증금을 계산한다.
- `useGlossary.js`: `/api/v1/terms` 호출, 로컬 fallback, 카테고리, 검색어, 필터링을 관리한다.
- `map/service.js`: 인프라 브리핑 데이터, 사용자 태그, 카카오맵 키 선택 로직을 제공한다.
- `useReportLibrary.js`: 마이페이지 탭, 다중 선택, 삭제 확인, 삭제, 실행 취소, 원래 정렬 복구를 관리한다.
- `shared/api/client.js`: 환경변수와 JWT accessor를 적용한 공통 API 인스턴스를 생성한다.

목업 데이터는 `legacy/data.js`에서 다음 모듈로 이동했다.

```text
frontend/src/mocks/ziptData.js
```

### 9. `window.*` 호환 레이어 제거

구조 개편 마지막 단계로 모듈 로딩 순서에 의존하던 전역 호환 코드를 제거했다.

제거한 전역:

- `window.React`
- `window.ReactDOM`
- `window.ZIPT_DATA`
- `window.ZIPT_TERMS`
- `window.ZIPT_API`
- `window.ZIPT_API_BASE`
- `window.ZIPT_KAKAO_MAP_KEY`
- `window.ZIPT_MOCKS`
- `Object.assign(window, ...)` 방식의 UI/Tweaks 등록

Tweaks 패널은 다음 모듈로 이동했다.

```text
frontend/src/shared/tweaks/TweaksPanel.jsx
```

앱 조립 코드도 `legacy/app.jsx`에서 다음 위치로 이동했다.

```text
frontend/src/app/LegacyApp.jsx
```

파일명에는 기존 화면 조립 역할을 나타내기 위해 `LegacyApp`을 유지했지만, 더 이상 `window.*` 전역이나 동적 import 순서에 의존하지 않는다.

최종 `main.jsx`는 전역 설정 없이 Router를 렌더링하는 역할만 담당한다.

```javascript
import ReactDOM from "react-dom/client";
import AppRouter from "./app/router.jsx";
import "./styles/tokens.scss";

ReactDOM.createRoot(document.getElementById("root")).render(<AppRouter />);
```

브라우저 기능에 필요한 `window.location`, `localStorage`, 카카오 SDK의 `window.kakao` 접근은 호환 레이어가 아니라 실제 브라우저 API이므로 유지했다.

### 11. 검증 결과

각 구조 변경 단계마다 Vite 프로덕션 빌드를 실행했다.

최종 검증 명령:

```powershell
cd frontend
.\node_modules\.bin\vite.cmd build
```

최종 결과:

```text
vite v7.3.5
68 modules transformed
build success
```

추가 확인 항목:

- `window.ZIPT_*` 참조 없음
- `window.React`, `window.ReactDOM` 참조 없음
- `Object.assign(window, ...)` 참조 없음
- 페이지와 Layout의 전역 React 주석 없음
- SCSS 컴파일 경고 없음
- React Router 빌드 오류 없음
- `legacy` 디렉터리의 구현 파일이 모두 모듈 구조로 이전됨

실제 브라우저 API와 외부 SDK를 위한 다음 접근은 정상 동작을 위해 유지했다.

- OAuth 리다이렉트용 `window.location`
- JWT 저장용 `localStorage`
- 카카오맵 SDK용 `window.kakao`
- Tweaks 호스트 통신용 `window.parent`, `postMessage`
