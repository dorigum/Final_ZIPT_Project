### 2. 분석 요청 mutation React Query 적용

**신규 파일:**

- `frontend/src/features/analysis/queries.js`
- `frontend/src/features/analysis/mutations.js`

**수정 파일:** `frontend/src/pages/analysis/AnalysisPage.jsx`

#### 2.1 queries.js

분석 이력과 상세 조회에 사용할 쿼리 키 팩토리와 `queryOptions`를 정의했다.

```javascript
export const analysisQueryKeys = {
  all: ["analysis"],
  history: () => ["analysis", "history"],
  detail: (id) => ["analysis", "detail", id],
};

export function analysisHistoryQueryOptions() { ... }
export function analysisDetailQueryOptions(id) { ... } // enabled: Boolean(id)
```

#### 2.2 mutations.js

`POST /api/v1/analysis` 호출을 `useMutation`으로 감쌌다. 분석 성공 시 `analysisQueryKeys.history()`를 무효화해 마이페이지 이력이 즉시 갱신되도록 했다.

```javascript
export function useAnalyzeMutation() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: (formData) => apiClient.analysis.analyzeRegistry(formData),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: analysisQueryKeys.history() });
    },
  });
}
```

#### 2.3 AnalysisPage.jsx 변경 사항

`Dropzone` 컴포넌트의 `onUpload()` 시그니처를 `onUpload(file)` 로 변경했다.

- 실제 파일이 전달될 경우: `FormData`를 생성해 mutation 호출 → 성공 시 API 결과를 `Report`에 전달, 실패 시 오류 화면 표시
- `null`이 전달될 경우 (샘플 등기부 버튼): 기존 demo 애니메이션 흐름 유지

페이지 phase에 `"error"` 상태를 추가하고, 오류 발생 시 `analyzeMutation.error.message`와 재시도 버튼을 표시한다.

`Report` 컴포넌트는 `apiResult` prop을 받아 실제 API 데이터가 있으면 그것을, 없으면 `getDemoAnalysisReport()` 데모 데이터를 표시한다.

### 3. 마이페이지 분석 이력 query React Query 적용

**수정 파일:** `frontend/src/features/mypage/useReportLibrary.js`

기존에는 `ziptData.myReports` 목업 데이터를 초기 state로 사용하고 있었다. `useQuery(analysisHistoryQueryOptions())`를 추가해 `GET /api/v1/analysis/history` 를 호출하고, 응답이 오면 목록을 교체한다. 백엔드가 실행되지 않거나 오류가 발생하면 목업 데이터로 자동 fallback한다.

응답 정규화 함수 `normalizeHistory(data)` 를 별도로 두어 백엔드가 `{ deeds, leases }` 형태 또는 flat 배열 형태로 응답하는 두 경우 모두 처리할 수 있도록 했다.

API 데이터가 새로 들어오면 (`source` 변경 감지) deeds/leases 로컬 state를 자동으로 교체한다. 선택·삭제·실행취소 등 UI 전용 상태는 기존과 동일하게 로컬 state로 관리한다.

훅이 반환하는 값에 `isLoading`, `isError`, `error`, `source` 를 추가해 상위 컴포넌트에서 상태별 화면을 분기할 수 있도록 했다.

### 4. 로딩·오류·빈 상태·인증 보호 화면 추가

#### 4.1 PageStates 공통 컴포넌트

**신규 파일:**

- `frontend/src/shared/components/PageStates.jsx`
- `frontend/src/shared/components/PageStates.module.scss`

세 가지 page-level 상태 컴포넌트를 작성했다.

| 컴포넌트 | 용도 | 주요 props |
|---|---|---|
| `PageLoading` | 스피너 + 메시지 | `message` |
| `PageError` | X 아이콘 + 메시지 + 재시도 버튼 | `message`, `onRetry` |
| `PageEmpty` | 인박스 아이콘 + 타이틀 + 설명 + 액션 | `title`, `description`, `action` |

`shared/components/index.jsx` 에 세 컴포넌트를 export해 어디서든 named import로 사용 가능하다.

#### 4.2 RequireAuth 인증 보호 컴포넌트

**신규 파일:** `frontend/src/app/RequireAuth.jsx`

`useAuthStore`에서 `isAuthenticated`와 `isInitializing`을 읽어 다음 세 가지 경우를 처리한다.

| 상태 | 동작 |
|---|---|
| `isInitializing === true` | `<PageLoading message="로그인 상태를 확인하는 중이에요…" />` 표시 |
| `isAuthenticated === false` | `<Navigate to="/" replace />` 로 홈으로 리다이렉트 |
| 인증됨 | `children` 렌더링 |

#### 4.3 App.jsx — 마이페이지 인증 보호 적용

**수정 파일:** `frontend/src/app/App.jsx`

```diff
- <Route path={ROUTE_PATHS.mypage} element={<MyPage />} />
+ <Route path={ROUTE_PATHS.mypage} element={<RequireAuth><MyPage /></RequireAuth>} />
```

미인증 상태에서 `/mypage`에 직접 접근하면 홈(`/`)으로 리다이렉트된다. 이전까지는 이 보호 장치 없이 누구나 마이페이지에 접근할 수 있었다.

#### 4.4 MyPage.jsx — 로딩·오류 상태 분기

**수정 파일:** `frontend/src/pages/mypage/MyPage.jsx`

`useReportLibrary`가 반환하는 `isLoading`, `isError`, `error`를 소비해 컴포넌트 상단에서 상태별로 다른 화면을 렌더링한다.

```javascript
if (isLoading) return <PageLoading message="분석 이력을 불러오는 중이에요…" />;
if (isError)   return <PageError message={error?.message} onRetry={() => window.location.reload()} />;
// ... 정상 화면
```

### 주제: — back_zipt → Mockup/backend 병합 작업


### 배경 및 목표

팀원들이 각 브랜치에서 개발한 백엔드 코드를 `back_zipt` 레포의 `test/backend-integration` 브랜치에 1차 병합해 둔 상태였음.
Aiden의 `Mockup/backend`에는 카카오 OAuth2, Flyway 마이그레이션, PostgreSQL 설정 등 인프라가 이미 갖춰져 있었고,
`back_zipt`에는 계약서 분석(Contract), 등기부 분석(Analysis), OCR, Spring AI 기능이 구현된 상태였음.

**결정 사항:** `Mockup/backend`를 베이스로 삼고 `back_zipt`의 기능을 이식한다.

  - 알 수 없는 URL은 `/`로 리다이렉트한다.

  - `<Outlet />`으로 페이지를 렌더링한다.
  - 페이지 이동 함수와 Tweaks 값을 Outlet context로 전달한다.
  - pathname 변경 시 스크롤을 상단으로 이동한다.

  - 페이지가 공통 `{ t, onGo }` context를 일관되게 읽도록 래퍼 hook을 추가했다.

### 2. 지도(인프라) 기능 모듈화

#### MapPage 축소

`frontend/src/pages/map/MapPage.jsx`는 약 580줄에 지도 SDK 로딩, 장소 검색, mock 지도, 카테고리 카드, 페이지 UI가 함께 있었다. 중복된 구형 Kakao 지도 래퍼를 제거하고 페이지 조합 역할만 남겨 약 150줄 수준으로 줄였다.

#### 분리 파일

```text
frontend/src/features/map/
├── InfraMap.jsx
├── InfraMap.module.scss
├── kakaoLoader.js
├── meta.js
└── components/
    ├── CategoryCard.jsx
    ├── CategoryCard.module.scss
    ├── MockInfraMap.jsx
    └── MockInfraMap.module.scss
```

- `InfraMap.jsx`: Kakao Maps SDK 지도 생성, 주소 지오코딩, 주변 POI 검색과 필터 적용
- `MockInfraMap.jsx`: API 키가 필요 없는 데모 지도 UI
- `CategoryCard.jsx`: 카테고리별 인프라 요약과 POI 목록
- `meta.js`: 인프라 카테고리와 사용자 persona 메타데이터
- `kakaoLoader.js`: `VITE_KAKAO_MAP_KEY`를 이용한 SDK 로딩

#### Kakao 키 관리

- JavaScript 키는 `frontend/.env.local`의 `VITE_KAKAO_MAP_KEY`로 설정했다.
- `.env.local`은 `frontend/.gitignore`에 의해 무시되는 것을 확인했다.
- Kakao REST API 키는 프론트 코드에 넣지 않았다. 향후 REST API 호출이 필요하면 백엔드 비밀 환경변수로 관리해야 한다.
- Vite는 시작 시 환경변수를 읽으므로 키를 변경한 후 개발 서버를 재시작해야 한다.

### 주제: — P0 분석 DTO 및 보증금 단위 계약 보강


### 1. 작업 목적

분석 화면은 사용자가 보증금을 `만원` 단위로 입력하지만 백엔드 분석 API와 데이터베이스는 `원` 단위를 사용한다. 단위 변환이 누락되면 사용자가 `24,000만원`을 입력했을 때 백엔드가 `24,000원`으로 해석하여 LTV, 위험 점수, HUG 보증보험 판정이 모두 잘못 계산될 수 있다.

또한 백엔드 `AnalysisResponse`의 필드명과 프론트 분석 리포트가 사용하는 화면 모델이 달라, API 응답을 그대로 렌더링할 수 없는 상태였다. 두 P0 항목을 기능 계층의 명시적인 변환 경계로 정리했다.

### 2. 분석 결과 DTO 화면 모델 보완

수정 파일:

- `frontend/src/features/analysis/normalizers.js`
- `frontend/src/pages/analysis/AnalysisPage.jsx`
- `frontend/src/features/analysis/__tests__/normalizers.test.js`
- `frontend/src/mocks/ziptData.js`

백엔드 응답에서 다음 필드를 화면 모델로 보존한다.

- 기본 정보: `address`, `propertyType`, `ownerName`
- 위험 정보: `riskScore`, `riskLevel`, `ltvRatio`, `totalLtv`
- HUG 정보: `hugDebtRatio`, `priorityRatio`, `insuranceEligible`, `insuranceConditions`
- 금액 정보: `officialPrice`, `marketPrice`, `deposit`, `totalPriorityAmount`
- 결과 파일: `pdfUrl`

화면 모델에서는 기존 UI가 사용하는 `safetyScore`, `appraisedValue`, `seniorDebt`, `flags`, `gabgu`, `eulgu` 구조를 유지한다. 숫자 필드는 유효한 숫자로 정규화하고 배열 응답이 `null`이어도 빈 배열로 처리한다.

분석 결과 카드에 고정되어 있던 `107%` 문구를 제거했다. 백엔드의 `hugDebtRatio`를 우선 사용하고, 값이 없으면 보증금과 선순위 채권을 인정 집값으로 나누어 계산한다. 집값이 0이거나 누락된 경우 `NaN` 또는 `Infinity`가 표시되지 않도록 0으로 방어한다.

샘플 데이터의 비율 문구도 실제 계산 반올림 결과인 `108%`로 통일했다.

### 3. 보증금 단위 변환 경계 통합

신규 파일:

- `frontend/src/features/analysis/request.js`
- `frontend/src/features/analysis/__tests__/request.test.js`

`createAnalysisFormData()`를 추가해 분석 요청 생성 책임을 기능 계층 한 곳으로 모았다.

입력과 전송 단위:

| 구간 | 필드 | 단위 | 예시 |
|---|---|---|---|
| 사용자 입력 | `depositManwon` | 만원 | `24,000` |
| API multipart 요청 | `deposit` | 원 | `240,000,000` |
| 백엔드/DB 저장 | `deposit` | 원 | `240,000,000` |
| 프론트 응답 모델 | `deposit` | 만원 | `24,000` |

변환 규칙:

```text
만원 → 원: 입력값 × 10,000
원 → 만원: 응답값 ÷ 10,000
```

기존에는 `AnalysisPage`와 개발용 `BackendSandboxPage`가 각각 직접 `FormData`를 만들고 단위를 변환했다. 두 화면 모두 `createAnalysisFormData()`를 사용하도록 변경하여 새 호출부에서 변환을 빠뜨릴 가능성을 줄였다.

요청 생성기는 음수 또는 숫자가 아닌 보증금을 거부한다. 테스트에서는 `24,000만원`이 multipart의 `deposit=240000000`으로 생성되는지 직접 검증한다.

### 4. 백엔드 API 단위 명시

수정 파일: `backend/src/main/java/com/zipt/analysis/controller/AnalysisController.java`

- Java 매개변수명을 `deposit`에서 `depositWon`으로 변경했다.
- HTTP 요청 필드명은 기존 계약을 유지하기 위해 `deposit`을 그대로 사용한다.
- Swagger `@Parameter` 설명에 `임차 보증금(원)`과 예시 `240000000`을 추가했다.
- 서비스 계층에는 원 단위 값만 전달한다.
