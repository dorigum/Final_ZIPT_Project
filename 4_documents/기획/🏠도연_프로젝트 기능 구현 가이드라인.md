## 1. 전세보증보험이란? (쉽게 이해하는 개념)
전세보증보험은 **ZIPT(집PT)** 서비스의 핵심 뼈대이자, 임차인들이 가장 돈을 많이 날리는 '전세 사기'를 막아주는 유일한 치트키입니다. 개념이 복잡해 보이지만 딱 **세 가지만** 이해하면 백엔드 로직으로 완벽하게 구현할 수 있어요.

도연님이 개발자 관점에서 알고리즘을 짤 수 있도록 핵심만 쏙쏙 골라 쉽게 풀어드릴게요! 🥲

> **"집주인이 돈 안 돌려주면, 나라(HUG)가 대신 갚아줄게!"**

전세 계약이 끝났을 때, 집주인이 "다음 세입자 안 들어와서 돈 못 줘!"라거나 갭투자로 파산해서 보증금을 안 돌려주는 경우가 많습니다. 이때 HUG(주택도시보증공사)가 세입자에게 보증금을 대신 먼저 돌려주고, 집주인에게 알아서 받아내는(구상권 청구) 보험 상품입니다.

따라서 임차인들은 "이 집이 HUG 보증보험에 가입이 되는 집인가?"를 계약 전에 무조건 확인하고 싶어 합니다. 가입만 되면 내 전 재산을 안전하게 지킬 수 있으니까요.

## 2. ZIPT 알고리즘에 넣어야 할 'HUG 가입 조건' 2가지

HUG는 아무 집이나 보험을 들어주지 않습니다. "이 집은 경매에 넘어가도 나라가 돈을 회수할 수 있겠다" 싶은 안전한 집만 가입시켜 줍니다. 백엔드에서 계산해야 할 **핵심 가입 기준 2가지**를 꼭 기억하세요.

### ① 담보인정비율 (집값 대비 빚 비율) ➔ 90% 이하만 가능!

- **개념:** 집값 중에서 (집주인의 대출 + 내 보증금)이 차지하는 비율입니다.
    
- **HUG 기준:** 이 비율이 **90% 이하**여야만 보험 가입이 가능합니다. (90%를 넘으면 나라에서 '깡통전세'로 보고 가입을 거절합니다.)
    
- **수식:**
    $$\text{담보인정비율} = \frac{\text{선순위 채권(은행 대출)} + \text{내 전세 보증금}}{\text{HUG가 인정하는 집값}} \times 100 \le 90\%$$

### ② 선순위 채권 비율 ➔ 60% 이하만 가능!

- **개념:** 내 보증금 말고, 집주인이 집을 담보로 은행에서 빌린 돈(등기부 을구의 근저당권)이 집값의 몇 %나 되는지 봅니다.
    
- **HUG 기준:** 은행 빚이 집값의 60%를 넘으면 보증보험 가입이 안 됩니다. 은행이 집을 먼저 뺏어가기 때문입니다.
    
- **수식:**
    $$\text{선순위 채권 비율} = \frac{\text{선순위 채권(은행 대출)}}{\text{HUG가 인정하는 집값}} \times 100 \le 60\%$$

## 3. 백엔드 개발자 입장에서 '데이터' 구하는 방법

방금 설명한 공식들을 코드로 구현하려면 "HUG가 인정하는 집값"을 어디서 가져올지가 중요합니다. HUG는 네이버 부동산 호가를 믿지 않고, 딱 정해진 공공데이터만 인정합니다.

### 🏢 아파트 / 오피스텔의 경우

1. **KB시세 또는 한국부동산원 시세:** 오픈 API나 크롤링을 통해 해당 매물의 시세를 가져옵니다. (HUG는 이 시세의 100%를 집값으로 인정해 줍니다.)
    
2. 만약 시세가 없다면 아래의 빌라 방식을 따릅니다.
    

### 🏡 빌라 / 다세대 주택의 경우 (전세 사기가 많은 곳!)

1. 국토교통부의 **'공시가격'** 오픈 API를 호출해 해당 매물의 올해 공시가격을 가져옵니다.
    
2. **산산 공식:** HUG는 공시가격의 140%를 집값으로 인정합니다.
    
    - _예시:_ 어떤 빌라의 공시가격이 1억 원이라면, HUG가 인정하는 이 집의 값은 $1\text{억} \times 1.4 = 1\text{억 } 4,000\text{만 원}$이 됩니다.
        
3. 백엔드에서는 이 $1\text{억 } 4,000\text{만 원}$을 기준으로 위의 **90% 공식**을 대입해 보증보험 가입 여부를 연산하면 됩니다.
    

## 💻 ZIPT 서비스에서 로직이 돌아가는 시나리오

1. **유저 정보 입력:** 유저가 계약하려는 집 주소와 본인의 '전세 보증금(예: 1억 2천만 원)'을 입력합니다.
    
2. **AI 서류 해독:** AI(ZIPT)가 등기부등본 을구에서 은행 빚(**'근저당권설정액: 2,000만 원'**)을 쏙 뽑아냅니다.
    
3. **공공데이터 조회:** 백엔드가 주소를 가지고 공시가격 API를 쳐서 '공시가격: 1억 원'을 가져옵니다.
    
4. **백엔드 연산:**
    
    - HUG 인정 집값 = $1\text{억} \times 140\% = 1\text{억 } 4,000\text{만 원}$
        
    - 총 부채 = 은행 빚(2,000만) + 내 보증금(1억 2천만) = **1억 4,000만 원**
        
    - 담보인정비율 = $\frac{1\text{억 } 4,000\text{만}}{1\text{억 } 4,000\text{만}} \times 100 = 100\%$
        
5. **최종 안전 보고서 출력:**
    
    > 🚨 **ZIPT 위험 경고!**
    > 
    > "도연님, 이 매물은 담보인정비율이 100%로 **HUG 기준(90%)을 초과**합니다. 전세보증보험 가입이 **불가능**한 깡통전세 매물이니 계약에 극도로 주의하세요!"

## 🛠️1. 부동산 서류 AI 분석 및 요약 기능 구현 가이드

이 기능의 핵심은 "AI(LLM)에게 너무 많은 것을 한 번에 시키지 않고, 필요한 데이터만 쏙쏙 뽑아내게 만드는 것"입니다.

### 📋 백엔드 개발 순서 (Data Flow)

1. **OCR 처리:** 유저가 업로드한 등기부등본(PDF/이미지)을 **Upstage나 Naver CLOVA OCR API**로 전송해 전체 텍스트를 추출합니다.

2. **비식별화 (보안):** 주민등록번호, 임대인 이름 뒷글자 등을 정규표현식으로 `***` 처리합니다.

3. **AI 데이터 파싱 (1차 GPT 호출):** 추출된 대량의 텍스트를 GPT API에 주입하며, "오직 아래 정보만 찾아서 구조화된 JSON으로 반환해"라고 요청합니다.
   
```json
{
  "owner_name": "홍*동",
  "has_collateral": true, 
  "collateral_amount": 120000000, 
  "has_danger_keywords": ["가압류", "신탁"],
  "auction_history": false
}
```

4. **백엔드 자체 연산:** AI가 뽑아준 `collateral_amount(은행 빚: 1억 2천)`과 유저가 입력한 `보증금`을 더한 뒤, 시세 API 데이터와 비교해 **부채 비율을 수학적 코드로 계산**합니다. (AI에게 계산을 시키면 높은 확률로 틀립니다!)

5. **최종 리포트 생성 (2차 GPT 호출):** 계산된 부채 비율 숫자와 위험 키워드 목록을 다시 GPT에게 던져주며, "이 데이터를 바탕으로 사회초년생이 이해하기 쉽게 번역한 용어 사전과 리포트를 작성해줘"라고 요청합니다.


### 💡 임차인 맞춤형 정보 수집 팁

- **전세 사기 다발 및 보증금 미반환 사례:** 주택도시보증공사(HUG) 등에서 제공하는 **'안심전세포털'의 악성 임대인 명단이나 자치구별 전세사기 통계 데이터**를 매주 한 번씩 내 DB에 미리 긁어다 놓습니다(Data Seeding). 그 후, AI가 추출한 임대인 정보나 주소지와 대조하는 로직을 백엔드에 추가하면 됩니다.

## 📖2. [용어 정리ZIP] 기능 구체화 및 뎁스(Depth) 확장 가이드

이 기능은 단순한 '단어장'이 아닙니다. 유저가 마주하는 상황(서류 읽을 때, 계약서 쓸 때)에 맞춰 **AI가 실시간으로 어려운 법률 용어를 임차인 맞춤형 언어로 리번역해 주는 인터랙티브 기능**으로 확장합니다.

### 1단계: 용어 사전의 데이터 구조화 (기초 뎁스)

먼저, 서비스의 뼈대가 될 필수 부동산 용어들을 백엔드 데이터베이스(DB)에 미리 정형 데이터로 적재해 둡니다.

- **카테고리 분류:** 용어를 찾기 쉽게 분류합니다.
    - `서류(등기부/대장)`: 근저당권설정, 가압류, 임차권등기명령, 위반건축물 등
        
    - `계약/권리`: 대항력, 확정일자, 우선변제권, 전입신고 등
        
    - `금융/보증`: HUG 보증보험, 선순위채권, 담보인정비율 등
        
- **3단계 번역 구조 (구체화 포인트):** 단어를 클릭했을 때 3개의 뎁스로 정보를 제공합니다.
    1. **원어 (법률 용어):** 근저당권설정
        
    2. **ZIPT 번역 (쉬운 표현):** "집을 담보로 집주인이 은행에서 빌린 돈"
        
    3. **임차인 행동 지침 (Action):** "이 금액과 내 보증금을 합친 금액이 집값의 60~70%를 넘으면 위험해요! 계약서 특약에 '잔금 시 대출 말소 조건'을 넣거나 다른 집을 알아보세요."
        

### 2단계: 서류 분석과 연동되는 '하이퍼링크(툴팁) 번역' (심화 뎁스)

유저가 등기부등본 분석 리포트를 볼 때, 리포트 텍스트에 포함된 어려운 단어들에 자동으로 하이퍼링크나 툴팁(마우스를 올리거나 터치하면 뜨는 말풍선)을 달아주는 기능입니다.

- **구현 가이드라인 (Data Flow):**
    1. AI(ZIPT)가 서류 분석 리포트 HTML/Markdown 텍스트를 생성합니다.
        
    2. 백엔드나 프론트엔드에서 텍스트 중 DB에 등록된 '부동산 용어 키워드'가 있는지 스캔합니다.
        
    3. 해당 키워드가 발견되면 자동으로 `<span class="word-tooltip" data-word="근저당권">근저당권</span>` 형태로 태그를 감싸줍니다.
        
    4. 유저가 리포트를 읽다가 모르는 단어를 터치하면, 화면이 이동하지 않고 그 자리에서 **"ZIPT 3초 번역" 말풍선**이 팝업으로 뜹니다. 이렇게 하면 유저가 서류를 읽는 흐름이 깨지지 않습니다.
        

### 3단계: 문장 통째로 번역하는 'AI 계약서 번역기' (최종 뎁스)

단어 검색을 넘어, 유저가 중개사에게 받은 문자 메시지나 특약사항 문장을 통째로 넣으면 AI가 숨겨진 의도까지 해독해 주는 기능입니다.

- **기능 시나리오:**
    - **유저 입력 문장:** "임차인은 목적물 인도 당일까지 임대인의 담보권 설정을 용인하기로 함." (흔히 쓰이는 독소 조항)
        
    - **ZIPT 번역 결과:**
        - ⚠️ **위험 조항 경고!**
            
        - **쉽게 말해서:** "내가 이사 가는 당일날, 집주인이 나 몰래 은행에서 집 담보로 대출을 받아도 눈감아주겠다는 뜻입니다."
            
        - **ZIPT의 조언:** "이 조항은 절대 동의하시면 안 됩니다! 내 보증금의 법적 순위가 은행 대출보다 뒤로 밀려 전세 사기 위험이 극도로 높아집니다. 반드시 '익일까지 담보권을 설정하지 않는다'로 수정을 고집하세요."

## 🗺️ 3. 초개인화 인프라 브리핑 기능 구현 가이드
카카오맵 API를 활용해 지도에 마커를 찍고, 이를 AI가 해석하게 만드는 세부 가이드입니다.
### 📋 기능 구현 순서 (Data Flow)

1. **주소 좌표 변환 (Geocoding):** 유저가 입력한 주소를 카카오 로컬 API(주소 검색 기능)에 던져서 해당 매물의 `위도(Latitude)`와 `경도(Longitude)` 좌표를 얻어냅니다.
    
2. **카테고리별 주변 시설 검색:** 카카오 로컬 API의 **'키워드로 장소 검색'** 또는 **'카테고리별 장소 검색'** API를 활용합니다. 매물 좌표를 중심점으로 잡고 `radius=1000` (반경 1km) 옵션을 주어 데이터를 요청합니다.
    - _카카오 카테고리 코드 활용:_ 지하철역(`SW8`), 편의점(`CS2`), 대형마트(`MT1`), 병원(`HP8`), 약국(`PM9`)
        
3. **도보 시간 계산 알고리즘:** 카카오 API가 준 매물과 시설 사이의 직선거리(미터 단위)를 사람이 걷는 평균 속도(분당 약 60~70m)로 나누어 "도보 X분"이라는 데이터로 가공합니다.
    
4. **AI 프롬프트 주입:** 가공된 인프라 데이터 리스트와 유저의 성향 키워드(`#뚜벅이`)를 AI에게 텍스트로 넘겨줍니다.

### 📝 AI에게 보낼 프롬프트 형태 예시


## 📄 1. 부동산 서류 AI 분석 및 요약 프롬프트

> **핵심 전략:** AI에게 복잡한 수치 계산(부채 비율 등)을 시키면 높은 확률로 수학적 에러(할루시네이션)를 냅니다. AI는 문서에서 **'위험 키워드와 핵심 숫자만 추출하는 파서(Parser)'** 역할을 수행하고, 줄글 리포트만 작성하도록 제한해야 합니다.

```text

[Tasks]
1. 갑구 분석: 압류, 가압류, 가처분, 신탁, 경매개시결정 등 소유권을 침해하는 위험 키워드가 있는지 확인하고 목록을 추출하세요.
```

## 🛠️ RAG를 적용할 3대 핵심 파이프라인 구조 스케치

팀 회의 내용을 바탕으로 백엔드에서 RAG 가이드라인을 잡으실 때 참고할 수 있는 데이터 흐름(Data Flow)입니다.
```TEXT
[1. 국토부 가이드 PDF / 법률 문서 / 지역 통계 API 데이터] │ ▼ (텍스트 추출 및 Chunking) [Embedding 모델] │ ▼ (저장) [Vector DB] (Chroma / Pinecone 등) ▲ │ (유저 매물 주소 및 등기부 리스크 기반 유사도 검색 Query) [2. 유저 매물 데이터] ┼───> [유사 문서 검색(Retrieve)] ──> [최적의 법률/통계 컨텍스트 확보] │ ▼ [3. 최종 화면 출력] <─── [정확하고 팩트에 기반한 답변 생성] <─── [OpenAI GPT-4o]
```

### 💡 실전 시나리오 예시 (특약사항 RAG 작동 방식)

1. **유저 상황:** 임차인이 올린 등기부에 '신탁' 위험 키워드가 발견됨.
    
2. **Retrieve (검색):** 시스템이 벡터 DB에서 '신탁 주택 계약 시 임차인 보호 표준 특약 조항'에 관한 법률 국토부 가이드라인 문서를 검색해 옴.
    
3. **Generation (생성):** GPT가 이 확실한 가이드라인 문서를 바탕으로 환각 없이 정확한 특약 문구를 생성: _"본 계약은 신탁회사의 사전 동의를 얻어 진행하며, 잔금은 신탁계좌로 입금한다"_ ➔ 유저에게 제공.


## 🚀 ZIPT(집PT)의 최종 기술 스택 한 줄 요약

팀원들과 기술 스택 정리하실 때 이 뼈대로 가시면 풀스택 포트폴리오로서 흠잡을 데가 없습니다.

- **Frontend:** Next.js (대시보드 및 지도 시각화, 모바일 웹)

- **AI & RAG:** LangChain / LlamaIndex + Vector DB + OpenAI API (서류 기반 특약사항/체크리스트/인프라 RAG 파이프라인)
    
- **Database:** PostgreSQL (유저 및 타임라인 데이터) + Redis (캐싱)
    
- **CI/CD:** Docker + GitHub Actions + AWS

## 🔍 1. 개발 착수 전, 팀원들과 '무조건' 명확히 해야 할 3가지

PM으로서 백엔드 및 AI 담당 팀원들과 아래 3가지를 명확히 짚고 개발을 시작해야 나중에 리워크(재개발)하는 사태를 막을 수 있습니다.

- **RAG 파이프라인의 데이터 포맷(Schema) 확정:**
    - "주변 소음/인프라 RAG"와 "특약사항 RAG" 결과물이 백엔드에서 프론트엔드로 넘어올 때, **어떤 JSON 구조**로 넘어올지 미리 문서화(Swagger 등)해야 합니다. 그래야 프론트와 백엔드가 동시에 병렬 개발을 진행할 수 있습니다.
        
- **파일 업로드 용량 및 확장자 제한:**
    - 임차인이 등기부등본을 올릴 때, PDF만 허용할지, 스마트폰으로 찍은 JPG/PNG 사진도 허용할지 결정해야 합니다. 이미지 변환이나 멀티파트(Multipart) 처리 로직이 프론트와 백엔드 모두에 영향을 주기 때문입니다.
        
- **외부 API 호출 제한(Rate Limit) 및 캐싱 정책:**
    - 카카오 로컬 API나 OpenAI API는 호출할 때마다 비용이 들거나 제한이 있습니다. 팀원들과 *"프론트에서 같은 주소를 재검색할 때는 백엔드가 Redis 캐시 데이터를 주도록 처리하자"*라는 합의를 미리 해두셔야 인프라 설계를 깔끔하게 가져갈 수 있습니다.
        
## 🎨 2. 프론트엔드 PM으로서 강력 추천하는 '프론트 협업 최적화 기능'
프론트엔드 개발자로서 기술적 역량(상태 관리, 최적화, 인터랙티브 UI)을 가장 화려하게 뽐낼 수 있으면서 AI 기능과 시너지가 나는 기능들입니다.
#### ① 실시간 스트리밍 UI (Streaming Response)
- **어떤 기능인가요?:** AI가 특약사항이나 인프라 브리핑 리포트를 작성할 때, ChatGPT처럼 텍스트가 한 글자씩 타다닥 출력되는 UI입니다.
    
- **프론트엔드 구현 포인트:** OpenAI의 Server-Sent Events(SSE)나 웹소켓 프로토콜을 프론트엔드에서 받아 실시간으로 상태(State)를 업데이트하며 끊김 없이 렌더링하는 기술을 보여줄 수 있습니다.
    
- **왜 좋을까요?:** AI 분석은 보통 5초~10초 이상 걸리는데, 그냥 로딩 스피너만 보여주면 유저가 이탈합니다. 스트리밍 UI는 **유저 경험(UX)을 극대화하는 동시에 프론트엔드의 비동기 데이터 처리 역량**을 가장 확실하게 증명합니다.
    

#### ② 드래그 앤 드롭 카드 인터페이스 기반의 '매물 비교 대시보드'
- **어떤 기능인가요?:** 임차인이 `수집.zip` 해놓은 매물(A집, B집)들을 한눈에 비교하는 화면입니다.
    
- **프론트엔드 구현 포인트:** 프론트엔드 라이브러리(예: `react-beautiful-dnd` 등)를 활용해, 유저가 마음에 드는 집 카드들을 '비교 영역'으로 드래그하여 드롭하면, 두 집의 **안전 점수, 인프라 도보 시간, 층간소음 통계 그래프**가 양옆(Side-by-Side)으로 나란히 비교 시각화되는 대시보드입니다.
    
- **왜 좋을까요?:** 화면 전환 없이 동적으로 데이터를 핸들링하는 프론트엔드의 '상태 관리(State Management)' 및 '컴포넌트 설계 능력'을 면접관에게 직관적으로 어필할 수 있습니다.
    

#### ③ '원터치 클립보드 복사' 기능이 포함된 AI 특약사항 카드
- **어떤 기능인가요?:** RAG를 통해 도출된 임차인 맞춤형 안심 특약사항 문구들을 컴포넌트화하는 것입니다.
    
- **프론트엔드 구현 포인트:** 특약사항 문구 우측 상단에 `[복사하기]` 버튼을 배치하고, 이를 누르면 토스트(Toast) 알림과 함께 클립보드에 복사되어 공인중개사나 임대인에게 카카오톡으로 바로 보낼 수 있게 만듭니다. 또한, 해당 특약이 왜 필요한지 숨겨진 의미를 보여주는 **인터랙티브 툴팁(Tooltip) UI**를 구현합니다.
    
- **왜 좋을까요?:** 디테일한 UX까지 섬세하게 신경 쓰는 PM으로서의 면모와, 깔끔한 UI 컴포넌트 분리 능력을 동시에 보여줄 수 있습니다.

### 데이터 정제
- 전/월세 계약 필수 핵심 용어 30개 추리기
- 나머지 데이터들은
	1. 위의 추천 사이트(특히 **찾기쉬운 생활법령정보**)에서 '주택임대차' 필수 단어 30개의 화면을 켭니다.
    
	2. 텍스트를 쭉 긁어서 하나의 메모장이나 노션에 붙여넣습니다.
    
	3. 아까 정한 **ZIPT 용어 번역기 AI 프롬프트**에 이 텍스트들을 밀어 넣고, 다음과 같이 요청합니다.
    > _"이 국가 공인 부동산 용어 설명을 읽고, 우리 DB 형식에 맞게 `{"word": "용어", "easy_definition": "3초 요약", "action_tip": "행동 가이드"}` 형태의 JSON 배열로 정제해줘."_
    
	4. 추출된 깔끔한 JSON 데이터를 백엔드 DB에 시드 데이터(Seed Data)로 쿵 찍어 넣으면 끝납니다.

#### PDF 데이터 정제
##### 1단계: PDF에서 날것의 텍스트만 추출하기 (Parsing)
파이썬에는 PDF 파일 안의 글자들을 그대로 긁어모아 주는 훌륭한 오픈소스 라이브러리들이 많습니다. (대표적으로 `PyPDF2`, `pdfplumber`, `LangChain의 PyPDFLoader` 등이 있습니다.)

- **어떻게 작동하나요?:** 파이썬 코드가 수백 페이지짜리 가이드북 PDF를 단 2~3초 만에 훑으며 안에 있는 모든 텍스트를 하나의 거대한 문자열(String)로 뽑아냅니다.
    

##### 2단계: 문맥 단위로 글자 쪼개기 (Chunking)
거대한 텍스트 통째로 DB에 집어넣으면 유저가 검색할 때 원하는 부분만 정확하게 보여줄 수 없습니다. 그래서 의미 있는 단락(Paragraph) 단위나 글자 수(예: 500자~1,000자씩) 단위로 텍스트를 숭덩숭덩 가위질해 주는 작업을 합니다.

- **어떻게 작동하나요?:** 이 역시 개발자가 손으로 쪼개는 게 아니라, 텍스트 분할기(Text Splitter) 코드를 돌리면 알아서 깔끔한 단락 배열(List)로 쪼개집니다.
    

##### 3단계: AI(GPT API)를 활용한 자동 JSON 데이터셋 변환 (치트키 🌟)
쪼개진 텍스트 단락들을 내 DB 양식에 맞게 정제하는 작업이 가장 까다로운데요, 여기서 **OpenAI API를 한 번만 딱 돌려주면 끝**납니다.

- **어떻게 작동하나요?:** 쪼개진 텍스트 조각들을 GPT API에 대량으로 밀어 넣으면서(Batch 처리), *"아래 텍스트를 읽고 `용어이름`, `쉬운_설명`, `임차인_행동_지침`으로 분류된 JSON 데이터로 변환해줘"*라고 프롬프트를 보냅니다.
    
- **결과:** AI가 밤새 수백 장의 문서를 읽고 우리가 설계한 데이터베이스 테이블 구조에 완벽하게 맞는 Clean 데이터(JSON)를 알아서 생산해 줍니다. 백엔드 개발자는 이 JSON 파일들을 DB에 `INSERT`만 해주면 끝나는 거죠!

#### 데이터 정제 시 참고 사항
- **가이드북은 딱 1~2개만 선정하기 (선택과 집중):**
    - 국토교통부, 서울시 주거포털 등에서 가장 최신이고 정리가 잘 된 임차인 가이드북 PDF 딱 1~2개(약 30~50페이지 내외)만 타깃으로 잡으세요. 이것만으로도 서비스에 필요한 필수 용어와 전/중/후 지식 콘텐츠는 90% 이상 커버됩니다.
        
- **초기 데이터 적재(Data Seeding)는 딱 한 번만:**
    - 이 작업은 서비스를 배포하고 운영할 때 실시간으로 돌아가는 로직이 아닙니다. **개발 단계에서 딱 한 번만** 스크립트를 돌려 내 DB에 기초 체력처럼 든든하게 데이터를 채워놓으면 끝나는 작업(Seed Data)입니다. 배포 후에는 외부 API를 치지 않으니 비용도 들지 않아요.
        
- **정 안 되면 엑셀(CSV) 노가다 활용하기:**
    - 정말 필수적인 부동산 용어(예: 대항력, 확정일자, 근저당권 등) 약 20~30개는 팀원들과 구글 스프레드시트에 `용어`, `쉬운 번역`, `꿀팁` 컬럼을 만들어 직접 타이핑한 뒤, **CSV 파일로 내보내서(Export) DB에 한 번에 넣는 것**이 코드를 짜는 것보다 빠를 수 있습니다.
