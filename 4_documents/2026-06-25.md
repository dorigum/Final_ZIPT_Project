오전 9:20

맞아요, 도연. “추후 수정 가능성”을 진짜 열어둘 거면 `terms.json`을 운영 데이터의 원천으로 두는 구조는 안 맞습니다. JSON seed는 배포 산출물에 묶이고, 현재처럼 classpath에서 읽으면 데이터 수정이 코드 배포 흐름에 종속돼요.

제가 추천하는 방향은 이거예요.

**추천 구조**

`terms.json`은 “초기 샘플/개발 seed”로만 남기고, 실제 용어 사전은 DB를 원천 데이터로 관리하는 구조가 좋습니다.

흐름은 이렇게 잡으면 됩니다.

```
관리자/운영자
  -> Admin Term API
    -> real_estate_terms / term_aliases DB
      -> 사용자용 Term API
```

현재 사용자 API는 이미 DB를 조회하고 있어서 방향 자체는 나쁘지 않습니다. 바꿔야 할 부분은 “데이터를 어떻게 넣고 수정하느냐”예요.

**구체 설계안**

1. `terms.json` 역할 축소  
    `src/main/resources/seed/terms.json`은 최초 개발 환경용 seed 또는 테스트용 fixture로만 둡니다. 운영 데이터의 정본으로 보지 않습니다.
    
2. Admin API 추가  
    관리자만 호출 가능한 API를 만듭니다.
    

```
POST   /api/admin/terms
PUT    /api/admin/terms/{termId}
PATCH  /api/admin/terms/{termId}
DELETE /api/admin/terms/{termId}
GET    /api/admin/terms
```

사용자용 API는 현재처럼 유지합니다.

```
GET /api/terms
GET /api/terms/{id}
```

3. Entity에 수정 메서드 추가  
    지금 `RealEstateTerm.create(...)`만 있고 수정 메서드는 없습니다.  
    `update(...)`, `replaceAliases(...)`, `activate/deactivate(...)` 같은 도메인 메서드를 추가하는 게 좋습니다.
    
4. 삭제는 물리 삭제보다 soft delete 추천  
    용어 사전은 화면, 검색, 로그, 북마크, 추후 분석 데이터와 연결될 가능성이 있어서 바로 삭제보다 `is_active` 컬럼을 두는 게 안전합니다.
    

```
is_active
created_at
updated_at
created_by
updated_by
```

5. 운영 반영은 DB 변경으로 끝나게 만들기  
    Admin API로 DB를 수정하면 배포 없이 바로 사용자 API에 반영됩니다.  
    캐시를 붙이지 않는다면 즉시 반영이고, Redis/cache를 붙이면 수정 시 캐시 evict만 해주면 됩니다.

**Seed Loader는 어떻게 바꾸면 좋나**

운영에서는 seed loader를 꺼두는 걸 추천합니다.

예를 들면:

```
zipt:
  terms:
    seed-enabled: false
```

개발 환경에서는 true, 운영 환경에서는 false.

```
dev: terms.json으로 초기 데이터 채움
prod: seed loader 비활성화, DB/Admin API로만 관리
```

이렇게 해야 운영 DB가 JSON seed에 의해 의도치 않게 바뀌는 일을 막을 수 있습니다.

**조금 더 좋은 확장안**

나중에 “수정 이력”이 중요해질 수 있으면 `term_revisions` 테이블을 따로 둘 수 있습니다.

```
real_estate_terms

- name
- category
- easy_definition
- action_tip
- risk_level
- official_definition
- source_name
- source_url
- license

- is_active
- created_at

term_aliases

term_revisions
- id
- term_id
- snapshot_json
- changed_by

```

당장 MVP라면 revision까지는 과할 수 있고, Admin API + soft delete + seed 비활성화 정도면 충분합니다.

1. `TermSeedDataLoader`를 `zipt.terms.seed-enabled` 옵션으로 제어
2. 운영 `application-prod.yml`에서는 seed 비활성화
3. `RealEstateTerm`에 수정 메서드 추가
4. 관리자용 `TermAdminController`, `TermAdminService` 추가
5. 사용자용 `TermService`는 active 용어만 조회하도록 변경
