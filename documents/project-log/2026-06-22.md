### 주제: back_zipt 병합 후 P1 버그 수정 및 누락 페이지 추가


### 개요

팀원이 작업한 back_zipt 코드를 Mockup/backend 로 병합하는 과정에서 발견된 프론트-백엔드 불일치 항목을 수정했다. 우선순위가 높은 P1 항목 4건을 처리하고, 병합 과정에서 빠진 페이지(로그인, 회원가입, 계약서 분석 결과)를 신규 추가했다.

### 1. [P1] 분석 이력이 전부 빈 목록으로 반환되는 문제 수정

**수정 파일:**

- `frontend/src/features/analysis/normalizers.js`
- `frontend/src/features/mypage/useReportLibrary.js`

#### 원인 분석

백엔드 AnalysisHistoryResponse DTO 에는 kind 또는 type 필드가 없다. 기존 normalizeAnalysisHistory() 함수는 입력이 배열이 아니거나 null 이면 null 을 반환했다. 이 null 이 useReportLibrary 의 useEffect 조건 `if (!apiData) return` 에 걸려 deeds 와 leases 상태가 갱신되지 않았고, 결과적으로 항상 초기 mock 데이터가 유지되거나 빈 목록이 표시됐다.

실제 데이터 흐름:

```
백엔드 GET /api/v1/analysis/history
  -> ApiResponse<List<AnalysisHistoryResponse>> 반환
  -> httpClient 가 payload.data 언래핑
  -> 배열 반환
  -> normalizeAnalysisHistory(배열) 호출
  -> { deeds: [...], leases: [] } 변환
  -> useReportLibrary 의 apiData 로 전달
```

#### 변경 내용

normalizers.js 변경 요약:

| 항목 | 이전 | 이후 |
|---|---|---|
| null/undefined 입력 | null 반환 | { deeds: [], leases: [] } 반환 |
| 비배열 입력 | null 반환 | 빈 배열로 처리 후 정규화 |
| 이미 정규화된 구조 감지 | data.deeds 또는 data.leases (falsy 위험) | data.deeds !== undefined 또는 data.leases !== undefined |

백엔드 DTO 에 kind 필드가 없으므로 배열 전체를 deeds(등기부 분석 이력)로 매핑하는 기존 로직은 유지했다.

useReportLibrary.js 변경 요약:

- source 판단 기준: `apiData ? 'api' : 'local'` -> `apiData?.deeds !== undefined ? 'api' : 'local'`
- useEffect 조기 종료 조건: `if (!apiData) return` -> `if (apiData?.deeds === undefined) return`

API 성공 후 이력이 0건인 경우(deeds: [])도 올바르게 화면에 반영되도록 수정했다.

### 3. [P1] 계약서가 항상 회원 ID 1번으로 저장되는 문제

**확인 파일:** `backend/src/main/java/com/zipt/contract/controller/ContractController.java`

이전 병합 커밋(552cef3)의 ContractController 에는 `Long memberId = 1L;` 하드코딩이 있었다. 최신 커밋(a7f9610)에서 이미 `Long memberId = resolveMember(authentication).getId();` 로 수정된 것을 확인했다.

resolveMember() 메서드 동작:

1. authentication 이 null 이거나 getName() 이 null 이면 UNAUTHORIZED 예외를 던진다.
2. JWT subject 가 숫자 형식이면 memberRepository.findById() 로 조회한다.
3. JWT subject 가 이메일 형식이면 memberRepository.findByEmail() 로 조회한다.
4. 회원이 없으면 MEMBER_NOT_FOUND 예외를 던진다.

MemberRepository 에 findByEmail(String email) 메서드 존재 확인 완료. 추가 조치 없이 현재 상태를 유지했다.

member/api.js 변경: 3개 엔드포인트 JSDoc 주석 문서화. /auth/signup 이 아닌 /auth/register 가 올바른 경로임 명시. refresh/logout/update 엔드포인트가 없으므로 logout 은 클라이언트 토큰 삭제 방식으로 처리함 명시.

- loginWithEmail({ email, password }): 로그인 성공 시 JWT 저장 -> initialize() 로 회원 정보 비동기 초기화. 실패 시 { ok: false, error } 반환
- registerWithEmail({ email, password, name }): 성공 시 { ok: true } 반환. 실패 시 { ok: false, error } 반환
- signOut: 서버 logout 엔드포인트가 없으므로 클라이언트에서 토큰만 삭제하는 기존 방식 유지. 주석으로 이유 명시

### 주제: back_zipt 원격 dev 브랜치 머지 및 front_zipt/backend 코드 이식


### 개요
`back_zipt` 원격 레포지토리의 최신 개발 내역을 가져와 머지 및 충돌 해결을 진행하고, 사용자의 프론트엔드가 참조할 수 있도록 관련 백엔드 개선 로직들을 `front_zipt/backend`로 성공적으로 이식 및 검증 완료했다.

---

### 세부 작업 내역

1. `back_zipt 원격 dev 브랜치 머지 및 충돌 해결` : back_zipt 프로젝트에 대해 원격 `dev` 브랜치의 32개 신규 커밋을 가져와 로컬 `Feature/OAuth` 브랜치에 머지했다. 머지 중 발생한 build.gradle, SecurityConfig, Member, MemberRepository, MemberController, application.yml 파일의 충돌을 dev 최신 OAuth2 구조에 맞춰 해결하고 커밋을 완료했다.
2. `로컬 변경사항(Stash) 적용 및 2차 충돌 해결` : 머지 직전 stash에 보관한 사용자의 로컬 변경사항들을 `git stash pop`으로 적용하였으며, 이때 발생한 9개 파일(build.gradle, SecurityConfig, AnalysisController, application-dev/prod.yml 등)의 충돌을 사용자의 로컬 동작(회원 ID 기반 위험진단, 로컬 .env 설정 등)이 유효하도록 완벽하게 수동 병합 및 staging 처리했다.
3. `front_zipt/backend로의 백엔드 개선 기능 이식` : back_zipt에 머지된 백엔드 변경사항들을 프론트엔드가 참고하는 `front_zipt/backend` 프로젝트에 성공적으로 이식했다. build.gradle의 Spring Boot/Spring AI 버전 상향 및 의존성 추가, Member 엔티티의 Lombok 최적화, AnalysisController/Service의 Long 회원 ID 기반의 깔끔한 API 연동, MarketPriceService/LtvService의 추가 법정동 코드 반영 및 로직 정교화를 이식했다.
4. `이식 코드 컴파일 빌드 검증 완료` : 이식 작업 완료 후 `front_zipt/backend` 디렉터리에서 `.\gradlew.bat compileJava`를 실행하여 컴파일 에러 없이 `BUILD SUCCESSFUL`로 정상 빌드되는 것을 검증 완료했다.
5. `3사 소셜 로그인(구글/카카오/네이버) 통합 이식` : 기존 카카오 단일 로그인 처리에서 구글, 카카오, 네이버를 통합 처리하는 `CustomOAuth2UserService` 구조로 업그레이드하고 관련 UserInfo 매퍼 및 팩토리를 `com.zipt.auth.oauth` 하위에 생성했다.
6. `Refresh Token 저장 및 재발급(reissue) 레이어 구축` : Refresh Token을 DB에 안전하게 보관하기 위한 `Auth` 엔티티와 리포지토리를 생성하고, `AuthService`에 `issueTokens` 및 `reissue` 메소드를 이식했다. `AuthController`에 `/api/v1/auth/refresh` API를 열어 쿠키 기반 리이슈가 가능하도록 했다.
7. `Flyway 마이그레이션 DDL 추가` : Refresh Token을 담을 `auth` 테이블을 생성하기 위해 `V7__create_auth.sql` 마이그레이션 스크립트를 작성하여 H2와 PostgreSQL 호환 테이블 생성을 완료했다.
8. `이식 코드 결합 빌드 검증 성공` : 사용하지 않는 기존 `KakaoOAuth2UserService.java`는 충돌 방지 및 Clean-up을 위해 파일 시스템에서 삭제하였으며, `.\gradlew.bat compileJava`를 통해 빌드가 정상 동작(`BUILD SUCCESSFUL`)하는 것을 검증 완료했다.
9. `SecurityConfig 순환 참조 문제 해결` : PasswordEncoder 빈을 별도의 PasswordEncoderConfig 구성 클래스로 분리하여 SecurityConfig -> CustomOAuth2UserService -> MemberServiceImpl -> PasswordEncoder -> SecurityConfig로 이어지던 BeanCurrentlyInCreationException 순환 참조 구조를 해제했다.
10. `OAuth2 테스트 로드용 설정 보완` : application.properties 파일에 정의되어 있던 google, naver 등의 클라이언트 ID/Secret 환경 변수에 기본값(dummy)을 제공하여, 로컬 테스트(gradle test)나 로컬 빌드 시 누락된 환경변수로 인해 애플리케이션 시작 과정에서 contextLoads 테스트가 실패하지 않도록 보완했다.
11. `전체 단위 및 통합 테스트 통과 검증` : `.\gradlew.bat test` 명령을 실행하여 프로젝트 내 모든 테스트가 성공적으로 통과(`BUILD SUCCESSFUL`)하며 정상 작동함을 컴파일 및 실행을 통해 최종 검증 완료했다.
12. `백엔드 테스트 슈트 이식` : `back_zipt` 프로젝트에만 등록되어 있던 핵심 단위 및 통합 테스트 클래스들(ActuatorSecurityTest, LtvServiceTest, MemberTest, JwtTokenProviderTest, AuthenticatedAnalysisFlowTest, ContractServiceTest)을 `front_zipt/backend` 환경에 맞게 패키지 구조 및 의존성을 조율하여 성공적으로 이식했다.
13. `이메일 로그인 토큰의 위험진단 연동 수정` : 이메일 로그인 성공 시 생성되는 email 기반 Access Token의 경우, `AnalysisController`에서 회원 번호 추출 시 파싱 에러(UNAUTHORIZED 401)가 발생하던 로직을 보완하여 이메일 주소로도 회원 정보를 조회할 수 있도록 변경해 연동 호환성을 확보했다.
14. `로컬 테스트용 Redis 헬스 체크 비활성화` : 로컬 개발 컴퓨터 및 테스트 환경 빌드 시 Redis 인스턴스 미구동으로 인해 Actuator 헬스 상태가 DOWN(503)이 되어 테스트가 실패하던 현상을 해결하기 위해 테스트 실행 프로퍼티에 `management.health.redis.enabled=false` 설정을 부여하여 테스트 안정성을 제고했다.
15. `보완된 전체 통합 테스트(14개) 통과 완료` : 새로 이식한 통합 및 단위 테스트를 포함하여 총 14개의 테스트가 모두 정상적으로 성공(`BUILD SUCCESSFUL`)함을 재검증 완료했다.
16. `프론트엔드 API 클라이언트 자동 토큰 재발급(Reissue) 및 재시도(Retry) 메커니즘 탑재` : Access Token 만료 시(401 Unauthorized 에러 발생), HttpOnly 쿠키에 담긴 Refresh Token을 사용해 백엔드의 `/api/v1/auth/refresh` API로 토큰 재발급을 요청하고, 성공 시 새 토큰을 로컬 스토리지에 저장한 뒤 원래 요청을 자동으로 다시 호출(Retry)하는 구조를 `httpClient.js`에 이식 완료했다.
17. `OAuth 로그인 콜백 파라미터 수신 호환성 개선` : 백엔드의 `OAuth2SuccessHandler`가 발급하는 `accessToken` 파라미터 명칭과 프론트엔드의 `token` 파라미터 명칭이 달랐던 부분을 보완하여, `OAuthCallback.jsx`에서 `token`과 `accessToken`을 모두 유연하게 인식하도록 수정했다.
18. `3사 소셜 로그인(구글/네이버) 프론트엔드 연동 지원` : `oauth.js`와 `useAuthStore.js`에 구글 및 네이버 로그인을 실행하는 액션을 추가하고 백엔드의 `/oauth2/authorization/{provider}` 엔드포인트와 매핑했다.
19. `개발 검증용 샌드박스 페이지 고도화` : `BackendSandboxPage.jsx` 파일에 카카오, 구글, 네이버 소셜 로그인 트리거 버튼과 로그아웃 버튼을 추가하고, 현재 토큰 보유 여부 및 Access Token의 실시간 적재 여부를 모니터링할 수 있는 UI를 구성했다.
20. `프론트엔드 프로덕션 빌드 검증 성공` : 수정된 `httpClient.js`, `OAuthCallback.jsx`, `BackendSandboxPage.jsx` 등을 포함하여 `npm run build`를 수행해 171개의 모듈이 에러 없이 빌드(`dist/assets`에 번들링) 완료되는 것을 검증했다.
21. `로컬 포트 충돌 해결 및 백엔드 서버 가동` : 기존에 8080 포트를 점유하고 있던 프로세스(PID 20072)를 확인하여 강제 종료(`taskkill`) 조치한 후, `.\gradlew.bat bootRun`을 통해 백엔드 서버를 8080 포트에서 정상 기동 완료했다.
22. `Flyway 스키마 마이그레이션 및 Swagger API 정상 동작 확인` : 서버가 가동되며 Flyway가 V1부터 V7(`create auth` 테이블 생성 포함)까지 7개의 마이그레이션을 에러 없이 안전하게 완료했음을 확인했으며, `http://localhost:8080/v3/api-docs`를 직접 호출하여 신규 인증 관련 엔드포인트가 노출된 Swagger API 스펙 JSON이 성공적으로 수신되는 것을 검증 완료했다.

### 세부 작업 내역

1. `back_zipt 원격 dev 브랜치 머지 및 충돌 해결` : back_zipt 프로젝트에 대해 원격 `dev` 브랜치의 32개 신규 커밋을 가져와 로컬 `Feature/OAuth` 브랜치에 머지했다. 머지 중 발생한 build.gradle, SecurityConfig, Member, MemberRepository, MemberController, application.yml 파일의 충돌을 dev 최신 OAuth2 구조에 맞춰 해결하고 커밋을 완료했다.
2. `로컬 변경사항(Stash) 적용 및 2차 충돌 해결` : 머지 직전 stash에 보관한 사용자의 로컬 변경사항들을 `git stash pop`으로 적용하였으며, 이때 발생한 9개 파일(build.gradle, SecurityConfig, AnalysisController, application-dev/prod.yml 등)의 충돌을 사용자의 로컬 동작(회원 ID 기반 위험진단, 로컬 .env 설정 등)이 유효하도록 완벽하게 수동 병합 및 staging 처리했다.
3. `front_zipt/backend로의 백엔드 개선 기능 이식` : back_zipt에 머지된 백엔드 변경사항들을 프론트엔드가 참고하는 `front_zipt/backend` 프로젝트에 성공적으로 이식했다. build.gradle의 Spring Boot/Spring AI 버전 상향 및 의존성 추가, Member 엔티티의 Lombok 최적화, AnalysisController/Service의 Long 회원 ID 기반의 깔끔한 API 연동, MarketPriceService/LtvService의 추가 법정동 코드 반영 및 로직 정교화를 이식했다.
4. `이식 코드 컴파일 빌드 검증 완료` : 이식 작업 완료 후 `front_zipt/backend` 디렉터리에서 `.\gradlew.bat compileJava`를 실행하여 컴파일 에러 없이 `BUILD SUCCESSFUL`로 정상 빌드되는 것을 검증 완료했다.
5. `3사 소셜 로그인(구글/카카오/네이버) 통합 이식` : 기존 카카오 단일 로그인 처리에서 구글, 카카오, 네이버를 통합 처리하는 `CustomOAuth2UserService` 구조로 업그레이드하고 관련 UserInfo 매퍼 및 팩토리를 `com.zipt.auth.oauth` 하위에 생성했다.
6. `Refresh Token 저장 및 재발급(reissue) 레이어 구축` : Refresh Token을 DB에 안전하게 보관하기 위한 `Auth` 엔티티와 리포지토리를 생성하고, `AuthService`에 `issueTokens` 및 `reissue` 메소드를 이식했다. `AuthController`에 `/api/v1/auth/refresh` API를 열어 쿠키 기반 리이슈가 가능하도록 했다.
